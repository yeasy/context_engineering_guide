## 13.2 金融：风控知识库构建

### 13.2.1 场景背景

金融机构需要对企业客户进行尽职调查（Due Diligence）。分析师需要综合查看新闻、财报、法律诉讼、股权结构等多源异构数据，识别潜在风险。

### 13.2.2 上下文设计挑战

1.  **多源异构**：数据包含 PDF 表格、新闻网页、股权关系图。
2.  **时效性**：昨天的安全企业今天可能就爆雷，Context 必须包含最新信息。
3.  **冲突消解**：不同来源的信息可能矛盾（如财报与新闻报道不一致）。
4.  **合规前置**：企业尽调不能只做资料汇总，还要覆盖 KYC、CDD、受益所有人识别、制裁/PEP 筛查、可疑交易线索、持续监控、人工审批和模型风险治理。

### 13.2.3 解决方案

#### KYC/CDD 上下文边界

金融风控知识库应先定义客户尽调数据模型，再决定检索策略：

-   **客户身份**：注册信息、实际经营地、行业、账户用途和预期交易行为。
-   **受益所有人（BO）**：直接或间接所有权、控制权、高管/授权代表关系，以及无法确认时的升级流程。
-   **风险筛查**：制裁名单、PEP、负面新闻、司法诉讼、监管处罚和异常交易线索。
-   **持续监控**：客户信息变化、交易行为偏离、新闻突发事件和定期复核触发器。
-   **数据治理**：来源、时间戳、许可证、更新频率、权限范围和保留期限必须进入元数据。

#### 多模态上下文解析

*   **PDF 解析**：使用专门的 Parser 提取财报中的表格，转化为 Markdown 格式，保留行列表头结构。
*   **图片 OCR**：提取扫描件中的文字。

#### 动态知识切片

根据风险维度将知识库切分为不同“切片”：

*   `Slice_Legal`: 诉讼、仲裁记录。
*   `Slice_Financial`: 财务报表、审计报告。
*   `Slice_News`: 舆情新闻。

当 Agent 分析“财务风险”时，优先加载 `Slice_Financial` 和 `Slice_News`，减少无关噪音。

#### 矛盾仲裁机制

在 System Prompt 中植入仲裁逻辑：

```markdown
当发现数据冲突时：
1. 不直接用固定来源优先级覆盖冲突；先列出来源、发布时间、适用期间、法律效力和可能偏差。
2. 将冲突分类为：事实口径不同、时间不同、主体不同、来源可信度不足或疑似错误。
3. 对监管披露、审计报告、法院/监管文书、客户自报、媒体报道分别给出证据等级和局限。
4. 输出"待人工确认"项，不自动作出开户、授信、冻结或报送决定。
```

#### 模型风险治理

-   **用途限制**：模型输出是分析初稿和风险线索，不是最终 KYC/CDD 结论。
-   **验证与监控**：上线前验证召回率、误报率、漏报率、来源覆盖和偏见风险；上线后监控数据漂移、规则漂移和异常审批率。
-   **变更管理**：模型、提示词、检索阈值、名单源和评分规则变更要有版本、评审、回滚和影响评估。
-   **人工审批**：拒绝开户、风险评级调整、可疑活动报送等动作必须由授权人员复核。

### 13.2.4 效果

系统能够自动生成尽调初稿，缩短分析师的资料整理与初稿撰写时间，并提示隐藏在多源资料中的关联风险。最终结论仍需合规/风控人员复核，且必须保留证据链、审批记录和模型版本。
