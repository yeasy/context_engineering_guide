## 13.1 医疗：电子病历问答系统

### 13.1.1 场景背景

医生在诊疗过程中，需要快速从患者厚厚的历史病历中查阅既往史、用药情况和检查报告。传统的关键词搜索难以处理“患者最近一次血糖异常是什么时候？”这类语义查询。

### 13.1.2 上下文设计挑战

1.  **准确性要求极高**：医疗场景容错率极低，必须把幻觉风险作为可监控、可追溯、可升级处理的工程风险，而不是只靠提示词承诺“绝不出错”。
2.  **术语复杂**：包含大量缩写、同义词和专业术语。
3.  **时序敏感**：病情的演变与时间高度相关，必须严格区分“现在”和“过去”。
4.  **隐私与责任边界**：若涉及 HIPAA 适用场景，应按 PHI/ePHI 处理，区分 covered entity、business associate、授权用途、最小必要访问、日志留存与删除流程；系统只能辅助临床决策，医生保留最终判断权。

### 13.1.3 解决方案

#### 合规与数据边界（上线前置）

-   **PHI 数据流**：进入解析、检索、模型调用、日志、评估集和人工标注的数据都要标注 PHI/ePHI 分类；默认不把原始病历、完整提示词或完整回答写入日志。
-   **供应商与部署**：第三方模型、OCR、向量库和观测平台需要先完成 BAA/数据处理条款、训练使用限制、子处理方和删除能力审查；不满足条件时使用本地化或专有实例。
-   **访问控制**：按患者、科室、角色和治疗关系做 RBAC/ABAC 校验；检索层必须过滤无权访问的病历片段。
-   **审计与留存**：记录 trace id、用户、患者范围、来源文档 ID、模型版本、检索分数、错误码和审批结果；原文调试只能在审批、脱敏、TTL 和访问审计下短期启用。
-   **CDS 边界**：临床决策支持应提供信息、选项、证据和不确定性说明，不能替代医生判断；用于诊断、治疗指令或时间敏感决策时，需要单独评估医疗器械/软件监管要求。

#### 结构化病历重组（写入策略）

原始病历通常是非结构化的文本。在写入向量库之前，先进行结构化处理：

```json
{
  "date": "2025-10-12",
  "type": "检查报告",
  "content": "空腹血糖 7.8 mmol/L",
  "entities": ["空腹血糖", "7.8"],
  "standardized_code": "GLU"
}
```

**关键点**：引入医学知识图谱（KG），将自然语言映射到标准术语代码（如 ICD-10, LOINC）。

#### 混合检索策略（选择策略）

采用 **关键词检索 + 向量检索 + SQL 查询** 的三路召回：

*   **关键词**：精确匹配药物名（“二甲双胍”）。
*   **向量**：匹配语义描述（“降糖药”）。
*   **SQL**：处理时间查询（“最近三个月” -> `date > NOW() - 90d`）。

#### 说明书式上下文（隔离策略）

System Prompt 采用严格的“说明书”风格：

```markdown
你是一个严肃的医疗助手。遵循以下原则：
1. 只能依据 <context> 标签中的内容回答。
2. 如果上下文中没有提及，必须回答"未在病历中找到相关记录"，严禁编造。
3. 引用每一条结论的来源（如：[2025-10-12 门诊记录]）。
4. 输出必须包含"证据来源"和"不确定性/需医生确认"两栏。
5. 不给出最终诊断或治疗指令；高风险、证据不足或时间敏感问题升级给医生。
```

### 13.1.4 效果评估

通过构建覆盖典型问答场景的黄金数据集，对比优化前后效果（示意口径）：

*   **准确性与一致性**：相较基线有明显提升。
*   **幻觉风险**：在“仅依据病历回答”的约束下显著降低。
*   **临床可用性**：抽检回答是否逐条引用病历、指南或检查记录，是否明确“不足以判断”的情况。
*   **合规性**：抽检 PHI 最小化、权限过滤、日志脱敏、删除请求和人工升级记录。
