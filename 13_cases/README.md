# 第十三章 行业应用与架构案例

上下文工程在垂直领域的应用往往需要结合具体的行业知识（Domain Knowledge）。本章前四节精选医疗、金融、法律、教育四个典型行业的架构案例，最后再补充一个跨行业的全自主智能体架构示意，展示如何将上下文工程策略与行业需求、系统形态深度融合。

> **合规前置声明**：本章案例不能直接照搬到生产环境。医疗、金融、法律、教育等行业在上线前必须先完成适用法域识别、数据分类、供应商条款审查、权限边界、人工审批、审计日志、评估指标和退出条件设计。若系统处理 PHI/PII、金融客户信息、律师客户资料或未成年人教育记录，应默认采用最小必要访问、最小留存、加密、RBAC/ABAC、删除流程、审计追踪和人工最终责任人机制。

## 落地前检查清单

- **数据流**：明确哪些数据进入模型、向量库、日志、评估集和人工标注流程。
- **权限边界**：按用户、租户、角色、案件/客户/学生范围隔离上下文和检索结果。
- **供应商控制**：确认日志留存、训练使用、子处理方、跨境传输、删除和审计条款。
- **人工审批**：高风险建议只能进入辅助决策流程，不能绕过医生、分析师、律师、教师或合规负责人。
- **证据追溯**：每个结论应能追溯到来源文档、版本、时间戳和检索路径。
- **退出条件**：定义低置信度、证据缺失、权限不足、法规冲突和安全风险时的拒答或升级流程。
